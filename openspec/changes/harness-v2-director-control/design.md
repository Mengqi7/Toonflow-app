# Harness V2 Director Control - 设计说明

> **当前阶段**: 原型验证与方案重定向  
> **设计原则**: 第一阶段先完成 Harness 工程基础闭环，不在主控台直接调用 ComfyUI。ComfyUI 作为后续生成后端插件接入。

## Context

当前 Toonflow 已具备可复用资产：剧本 Agent、工作流编排、提示词与模型配置、Skills 管理、全局设置中心，以及剧本/资产/分镜/工作台等业务页面。旧版 Harness 主控台 demo 的主要问题是把界面重心放在生图网格和 ComfyUI 参数快照上，用户看到的是工具面板，而不是 AI 制片系统的指挥中心。

Harness V2 的目标不是替换 Toonflow 现有能力，而是在现有能力上嵌入闭环：导演 Agent 负责调度，多工种 Agent 负责执行，质量监制 Agent 负责自动审核，用户只在关键门禁做决策。

## Goals / Non-Goals

**Goals:**

- 建立 AI影视智能工作台的信息架构：导演中枢、任务图、工种 Agent、产物流转、审核门禁、版本历史、设置入口。
- 让工作台完整呈现从小说/剧本到成片的闭环：规划、执行、回写、审核、终审、打回、重试、推进。
- 将用户从执行层移动到决策层：用户可以暂停/恢复流程、修改指令、确认阶段终审、选择打回策略。
- 复用 Toonflow 设置中心能力，支持 Harness 下 Agent、模型、提示词、Skills、记忆配置的在线编辑入口。
- 先产出可验证原型，待产品逻辑确认后再继续正式开发。

**Non-Goals:**

- 第一阶段不在主控台展示 ComfyUI 参数编辑器、workflow 测试运行、直接切换 workflow 或直接调用 ComfyUI。
- 第一阶段不要求 ComfyUI 服务可用，也不把 Harness 基础闭环依赖在 ComfyUI 上。
- 不重写 Toonflow 已验证的业务页面；主控台优先做管控与回写视图，业务页面继续作为产物细节入口。
- 不做多人协同与复杂权限系统。

## Decisions

### 1. 工作台采用“三栏指挥舱”而不是生图网格仪表盘

主控台 SHALL 采用高密度工作台布局：

- 左侧：导演 Agent 对话与用户决策卡片。
- 中央：流程阶段、Agent 任务图、产物队列、当前执行状态。
- 右侧：待处理审核门禁、版本历史摘要、全局设置入口。

这样可以同时体现“AI 导演可对话”“Agent 在协作”“用户只处理关键决策”三件事。

### 2. Agent 分工升级为工业流水线工位体系

第一阶段原型和后续正式 UI SHALL 不再只展示 8 个粗粒度角色，而是按影视工业工序展示“调度类 + 执行类 + 审核类”三大体系。主控台左侧以流水线管线展示细分工位，中央展示选中工位的输入、过程、产物和审核闭环，右侧展示全局监控、打回链路和人工终审。

调度类 Agent：

- 总导演调度 Agent：接收人工制片需求，拆解任务、调度节点、处理异常链路、统一风格。

执行类 Agent：

- 剧本改编 Agent：小说/大纲转影视剧本，输出分场次剧本。
- 台词打磨 Agent：优化人物台词、情绪、语速与时序。
- 角色设定 Agent：输出角色设定卡、一致性约束与视觉标签。
- 服化道设计 Agent：输出服装、妆容、道具与风格参数。
- 场景概念 Agent：输出场景设定、空间布局、天气和光影参数。
- 衍生图合成 Agent：输出角色着装图、场景效果图、道具特写图及提示词/模型日志。
- 分镜脚本 Agent：输出逐镜头脚本、景别、运镜、机位、动作和时长。
- 分镜提示词优化 Agent：输出逐镜头正向/负向提示词、模型参数建议和一致性约束。
- 分镜生图执行 Agent：输出分镜成品图、模型调用日志、种子与参数。
- 视频生成 Agent：输出单镜头视频片段、动效提示词、参考图和模型参数。
- 一致性校验 Agent：输出跨镜头一致性报告与偏差片段标注。
- 配音生成 Agent：输出逐镜头配音音频、音色与语速参数。
- 配乐适配 Agent：输出配乐音轨、环境音效和情绪曲线映射。
- 剪辑合成 Agent：输出成片视频、剪辑工程文件和音画同步结果。

审核类 Agent：

- 剧本内容审核、角色设定审核、场景道具审核、分镜逻辑审核、提示词合规审核、图片质量审核、视频质量审核、音画同步审核、总监制 Agent。

每个 Agent 必须声明输入来源、输出产物、存储位、上游对接方、下游审核方和被打回时的重跑目标。所有产物必须携带生成 Agent、版本号、时间、调用模型、提示词、参考素材、审核记录和 retryInstruction，以实现问题可定位、责任可追溯、链路可回溯。

### 3. 多级审核是第一阶段核心交互

每个阶段产物生成后，先由质量监制 Agent 自动审核；未达标则根据 retryInstruction 自动打回对应工种。阶段完成时进入人工终审门禁，用户可选择：

- 通过并进入下一阶段
- 打回当前工种
- 跨工种打回上游
- 人工修改后再审
- 暂停流程并补充指令

审核结果与打回记录必须进入版本历史，便于对比、回滚和复盘。

### 4. ComfyUI 改为二期生成后端插件

第一阶段只保留生成后端抽象概念，不在主控台直接暴露 ComfyUI。后续版本可在模型服务或 Harness 工程管理中新增“生成后端插件”，再接入 ComfyUI server/workflow/parameter editor。

当前 spec 中所有 ComfyUI 直接调用、参数编辑、测试运行、workflow 切换、服务管理页面都视为二期范围。

### 5. 原型先行，开发后置

当前阶段必须先交付：

- Figma 多画面原型：主控台、审核门禁、Agent/全局设置、流程导览。
- 本地 HTML 动态原型：可点击切换场景，验证流程流转、审核门禁、Agent 分工和设置入口。
- OpenSpec 修订：明确 ComfyUI 延后、重置 UI 任务口径。

原型确认后再进入正式开发，开发时优先重做工作台体验，再继续接口和后端闭环验证。

## UI Prototype Structure

### 画面 1: 工厂主控 / Harness 制片流水线

- 顶部显示项目名、实例状态、暂停/恢复/版本入口。
- 左侧以发光管线展示调度、执行、审核三类 Agent 工位，并用状态色标识运行中、审核中、通过、打回重跑。
- 中央展示选中工位的四象限详情：输入信息区、工作过程区、产物输出区、审核闭环区。
- 右侧展示全局进度、任务统计、异常告警、资源占用、循环次数、待处理门禁和全局设置入口。

### 画面 2: 分镜生产线 / 重点透明化详情

- 横向时间轴展示镜头顺序、缩略图、镜号、景别、时长、审核状态。
- 单镜头详情展示完整分镜文字、镜头语言参数、生图提示词全文、负面提示词、模型名称、采样参数、参考素材和多版本对比。
- 批量操作展示提示词合规报告、批量重生成、批量调整一致性约束。

### 画面 3: 审核门禁 / 专项审核与人工终审

- 展示质量监制报告、评分维度、问题清单、推荐处理方式。
- 决策面板提供通过、打回原生产 Agent、跨工种打回、人工修改后再审。
- 明确每个决策会如何影响 DirectorOrchestrator 的下一步任务图。
- 展示“不通过 -> 结构化修正意见 -> 原路打回 -> 重启生产 Agent -> 再审”的 Loop 链路。

### 画面 4: 设置中心 / Harness Agent 在线配置

- 复用 Toonflow 全局设置入口。
- 展示模型服务、Agent 配置、提示词管理、Skills 管理、记忆配置、Harness 工程管理。
- 每个 Harness Agent 支持进入编辑 system prompt、skills、模型映射、retry 策略。
- 明确标注 ComfyUI 直接调用为二期预留。

### 画面 5: 流程导览 / 自驱闭环与追溯链

- 用闭环图展示输入、规划、执行、回写、内审、门禁、驳回、推进。
- 右侧显示实时事件流样例，包括 `task.started`、`callback.persisted`、`review.scored`、`user_input_required`、`review.reroute`、`version.created`。
- 展示产物 ID、生成 Agent、审核 Agent、版本链、模型、提示词、打回原因和重跑次数。

## Data Flow

1. 用户输入小说、目标或修改指令。
2. DirectorOrchestrator 解析意图并生成/更新 TaskGraph。
3. WorkflowRunner 执行单个工种任务。
4. Agent 输出产物，通过 CallbackBridge 写回 Toonflow 业务表。
5. HarnessEventBus 推送实时事件到主控台。
6. ReviewPipeline 和质量监制 Agent 自动审核产物。
7. 通过则进入阶段终审；不通过则生成 retryInstruction 并打回。
8. 用户在终审门禁做决策，DirectorOrchestrator 根据决策推进或重试。

## Risks / Trade-offs

- **原型与正式实现偏差**: 原型必须使用真实路由、真实表名和真实事件名，避免纯视觉稿失真。
- **Agent 角色过细导致 UI 复杂**: 第一阶段 UI 必须展示细分工位，但通过调度类、执行类、审核类分组和选中工位详情承载复杂度，避免把所有细节同时摊平成噪音。
- **ComfyUI 延后影响生图能力想象**: 原型保留“生成后端插件”入口，但不展示参数面板，避免误导当前范围。
- **现有前端半成品干扰判断**: 旧 HarnessControlRoom 只能作为技术草稿，正式开发以新原型为准。

## Migration Plan

1. 修订 OpenSpec，明确第一阶段不做 ComfyUI 直接调用。
2. 产出 Figma 和本地 HTML 原型，先验证信息架构与交互。
3. 原型确认后，重做 Toonflow-web 的 HarnessControlRoom。
4. 移除前端 ComfyUI 参数面板、测试运行按钮和直接调用逻辑。
5. 对接现有 SSE、TaskGraph、CallbackBridge、ReviewPipeline。
6. 完成浏览器烟测、构建验证和端到端关键路径验证。
