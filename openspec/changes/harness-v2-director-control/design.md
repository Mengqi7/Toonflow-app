## Context

当前 Toonflow 已具备可编排工作流、剧本 Agent、提示词与模型体系、Skills/Memory 配置、审核与版本记录等底座，但工作台 UI 仍将多个阶段混合在同一屏，导致“能做什么”和“怎么流转”不够清晰。用户希望 Harness 成为 AI 影视制作的操作内核，而不是另一个参数面板。

## Goals / Non-Goals

**Goals:**
- 建立清晰的三层产品结构：对话中枢、阶段工作台、产物仓库。
- 保留并强化 Harness 内核能力：调度、执行、审核、回写、版本化。
- 让每个阶段都可承载真实产物和可追溯记录。
- 参考 LaperAI 的优点，但形成 Toonflow 自己的流程语言和界面秩序。

**Non-Goals:**
- 第一阶段不接入 ComfyUI 直接调用面板。
- 不在一个屏里承载所有阶段的完整细节。
- 不重做 Toonflow 的全部业务页。
- 不在本次设计里引入新的外部后端依赖。

## Decisions

### 1. 用“三层结构”替代单屏大总览
对话中枢负责接收用户意图与确认，阶段工作台负责按流程展示当前工位，产物仓库负责保存每一步输出。  
备选方案是继续把所有信息压缩在单页里，但这会持续放大认知负担，也不利于产物沉淀。

### 2. Harness 内核采用“调度 / 执行 / 审核 / 回写”分层
DirectorOrchestrator 处理任务图与自动调度，WorkflowRunner 只执行单任务，ReviewPipeline 负责判断是否打回，CallbackBridge 负责把产物写回业务表和版本仓库。  
备选方案是让 WorkflowRunner 继续承担大部分流程编排，但这会让职责边界模糊，后续很难扩展。

### 3. 产物以“阶段资产”而不是“工具输出”来设计
剧本、人物、场景、道具、分镜、视频、音轨、审核意见都应成为可独立查看和版本化的对象。  
备选方案是把它们只作为日志或弹窗内容，但那样无法形成完整的影视生产链。

### 4. UI 借鉴 LaperAI，但不复制它的单页组织
借鉴其阶段化工作流、清晰的版本链、对话式操作入口和专业感；保留 Toonflow 的项目、工作流和设置体系。  
备选方案是 1:1 复刻外部产品，但既有版权与实现风险，也不符合我们已有底座。

### 5. 案例工程作为默认示范流
默认案例应展示“小说 -> 剧本 -> 人物 -> 场景道具 -> 分镜 -> 视频 -> 终审”的连续闭环，并允许用户在任意关键点插入修改。  
备选方案是只展示单阶段操作，但用户很难理解 Harness 的整体价值。

## Risks / Trade-offs

- [Risk] 三层结构初期会显得页面比单屏更分散 -> [Mitigation] 通过统一侧栏、阶段标签、版本轨和事件流保持上下文连续。
- [Risk] 产物仓库做重了会拖慢首屏 -> [Mitigation] 首屏只展示摘要和最近版本，详情按需展开。
- [Risk] 与 LaperAI 的相似度过高 -> [Mitigation] 只借鉴产品语言和工作流组织，不复制视觉细节与结构层级。
- [Risk] Harness 内核职责边界不清 -> [Mitigation] 明确 DirectorOrchestrator / WorkflowRunner / ReviewPipeline / CallbackBridge 的单一职责。

## Migration Plan

1. 先重构主控台信息架构与术语。
2. 再补阶段工作台的产物承载位与版本轨。
3. 然后梳理 Harness 内核职责边界和事件流。
4. 最后接回 Toonflow 现有业务页与回写链路。

## Open Questions

- 阶段之间是否允许用户自由跳过，还是必须经过终审门禁。
- 产物仓库是否需要独立筛选“案例工程产物”和“真实项目产物”。
- 首个案例工程应优先展示短剧、长剧，还是电影分镜式流程。
